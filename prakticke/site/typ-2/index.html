Link Aggregation — SW1 ↔ SW2
LACP / PagP / EtherChannel dle zadání
Realizujte Link Aggregation mezi SW1 a SW2. Zadání specifikuje protokol: LACP, PagP, nebo EtherChannel (ON). Port-channel nastavte jako trunk.
LACP — SW1 (active) + SW2 (passive) 
! SW1
SW1(config)# interface range GigabitEthernet0/1-2
SW1(config-if-range)# channel-protocol lacp
SW1(config-if-range)# channel-group 1 mode active
SW1(config-if-range)# exit
SW1(config)# interface Port-channel1
SW1(config-if)# switchport mode trunk

! SW2
SW2(config)# interface range GigabitEthernet0/1-2
SW2(config-if-range)# channel-protocol lacp
SW2(config-if-range)# channel-group 1 mode passive
SW2(config-if-range)# exit
SW2(config)# interface Port-channel1
SW2(config-if)# switchport mode trunk

! Ověření:
SW1# show etherchannel summary
// TIP — VLANy na SW1 a SW2 Nezapomeňte vytvořit VLANy a přiřadit access porty ještě před nastavením Link Aggregation. vlan 100 / name VLAN100 na obou switších.
d)
Routing — inter-VLAN + R1 ↔ R2
R1, R2 — statický i OSPF
Nakonfigurujte inter-VLAN routing (router-on-a-stick na R2) a routování mezi R1 a R2 oběma metodami: staticky i OSPF.
R2 — inter-VLAN + propojovací linky 
R2(config)# interface GigabitEthernet0/0
R2(config-if)# no shutdown
R2(config-if)# exit

! VLAN100
R2(config)# interface GigabitEthernet0/0.100
R2(config-subif)# encapsulation dot1Q 100
R2(config-subif)# ip address 10.1.0.1 255.255.255.0
R2(config-subif)# exit

! VLAN200
R2(config)# interface GigabitEthernet0/0.200
R2(config-subif)# encapsulation dot1Q 200
R2(config-subif)# ip address 10.1.1.1 255.255.255.0
R2(config-subif)# exit

! Propojovací linka R2 ↔ R1
R2(config)# interface GigabitEthernet0/1
R2(config-if)# ip address 10.1.2.66 255.255.255.252
R2(config-if)# no shutdown
R2(config-if)# exit

! R1 — strana R1-R2 linky + Internet
R1(config)# interface GigabitEthernet0/0
R1(config-if)# ip address 10.1.2.65 255.255.255.252
R1(config-if)# no shutdown
R1(config-if)# exit
R1(config)# interface GigabitEthernet0/1
R1(config-if)# ip address 213.200.13.X 255.255.255.0
R1(config-if)# no shutdown
OSPF — R1 + R2 
! R2 — všechny vnitřní sítě
R2(config)# router ospf 1
R2(config-router)# network 10.1.0.0 0.0.0.255 area 0
R2(config-router)# network 10.1.1.0 0.0.0.255 area 0
R2(config-router)# network 10.1.2.64 0.0.0.3   area 0
R2(config-router)# exit

! R1 — propojovací linka
R1(config)# router ospf 1
R1(config-router)# network 10.1.2.64 0.0.0.3 area 0
R1(config-router)# exit

PC1 ↔ PC3, PC2 ↔ PC4, PC → servery
Ověřte vzájemnou dosažitelnost všech zařízení v rámci COMPANY.local. Teprve po úspěšném pingu přistupte ke konfiguraci DHCP.
Test pingem z PC (Command Prompt) 
! Z PC1 (VLAN100) na PC3 (VLAN200) — inter-VLAN routing
C:\> ping 10.1.1.10

! Z PC1 na https server v DMZ
C:\> ping 10.1.2.2

! Na routeru — ověření routovací tabulky

R1# show ip route

! Trace route pro diagnostiku
C:\> tracert 10.1.2.2
// NEFUNGUJE PING? Zkontrolujte: (1) trunk port ke switchi, (2) sub-interface encapsulation, (3) no shutdown na všech rozhraních, (4) správná default gateway na PC.
f)
DHCP — R1 pro VLAN100 a VLAN200
Řešte až po funkčním pingu (úkol e)
Nakonfigurujte DHCP na R1 pro oba VLANy. Protože DHCP server (R1) je jiný router než gateway (R2), přidejte ip helper-address.
R1 — DHCP pooly + R2 helper-address 
! R1 — DHCP pooly
R1(config)# ip dhcp excluded-address 10.1.0.1 10.1.0.20
R1(config)# ip dhcp excluded-address 10.1.1.1 10.1.1.20

R1(config)# ip dhcp pool VLAN100
R1(dhcp-config)# network 10.1.0.0 255.255.255.0
R1(dhcp-config)# default-router 10.1.0.1
R1(dhcp-config)# dns-server 8.8.8.8
R1(dhcp-config)# exit

R1(config)# ip dhcp pool VLAN200
R1(dhcp-config)# network 10.1.1.0 255.255.255.0
R1(dhcp-config)# default-router 10.1.1.1
R1(dhcp-config)# dns-server 8.8.8.8
R1(dhcp-config)# exit

! R2 — přeposílání DHCP požadavků na R1
R2(config)# interface GigabitEthernet0/0.100
R2(config-subif)# ip helper-address 10.1.2.65
R2(config-subif)# exit
R2(config)# interface GigabitEthernet0/0.200
R2(config-subif)# ip helper-address 10.1.2.65

SSH — R1, přístup jen z jednoho VLANu
VLAN100 může, VLAN200 nemůže
SSH přístup na R1: z VLAN100 povoleno, z VLAN200 odepřeno.
R1 — SSH + ACL omezení 
R1(config)# hostname R1
R1(config)# ip domain-name company.local
R1(config)# crypto key generate rsa modulus 1024
R1(config)# ip ssh version 2
R1(config)# username admin privilege 15 secret heslo

! ACL — povoleno pouze z VLAN100
R1(config)# ip access-list standard SSH_VLAN100
R1(config-std-nacl)# permit 10.1.0.0 0.0.0.255
R1(config-std-nacl)# deny   any
R1(config-std-nacl)# exit

R1(config)# line vty 0 4
R1(config-line)# login local
R1(config-line)# transport input ssh
R1(config-line)# access-class SSH_VLAN100 in

NAT + Default route — R1
Pouze VLAN100 má přístup do Internetu
Na R1 nakonfigurujte NAT overload — přístup do Internetu pouze pro zařízení z VLAN100. Z VLAN200 přístup do Internetu není povolen. Nezapomeňte na default route a její distribuci k R2.
R1 — NAT + default route 
! Default route na Internet
R1(config)# ip route 0.0.0.0 0.0.0.0 213.200.13.1

! OSPF — distribuce default route k R2
R1(config)# router ospf 1
R1(config-router)# default-information originate
R1(config-router)# exit

! Označení rozhraní
R1(config)# interface GigabitEthernet0/0
R1(config-if)# ip nat inside
R1(config-if)# exit
R1(config)# interface GigabitEthernet0/1
R1(config-if)# ip nat outside
R1(config-if)# exit

! ACL — překládat pouze VLAN100
R1(config)# ip access-list standard NAT_VLAN100
R1(config-std-nacl)# permit 10.1.0.0 0.0.0.255
R1(config-std-nacl)# exit

! PAT
R1(config)# ip nat inside source list NAT_VLAN100
              interface GigabitEthernet0/1 overload
// STATICKÉ ROUTOVÁNÍ Pokud nepoužíváte OSPF, default route musíte přidat na R2 ručně: R2(config)# ip route 0.0.0.0 0.0.0.0 10.1.2.65

Zpřístupněte servery v DMZ z Internetu: https/443 na https server a FTP (20, 21) na ftp-log server.

! HTTPS → https server
R1(config)# ip nat inside source static tcp
              10.1.2.2 443 213.200.13.X 443

R1(config)# ip nat inside source static tcp
              10.1.2.3 21  213.200.13.X 21

l)
ACL — VLAN100 může HTTPS, VLAN200 nesmí
Extended ACL na R2 sub-interfaces
Z VLAN100 povolte přístup na https server (port 443). Z VLAN200 komunikaci na https server úplně zakažte.
R2 — Extended ACL 
! VLAN100 → HTTPS povoleno
R2(config)# ip access-list extended VLAN100_OUT
R2(config-ext-nacl)# permit tcp 10.1.0.0 0.0.0.255 host 10.1.2.2 eq 443
R2(config-ext-nacl)# permit ip  10.1.0.0 0.0.0.255 any
R2(config-ext-nacl)# exit

! VLAN200 → HTTPS zakázáno, ostatní povoleno
R2(config)# ip access-list extended VLAN200_OUT
R2(config-ext-nacl)# deny   tcp 10.1.1.0 0.0.0.255 host 10.1.2.2 eq 443
R2(config-ext-nacl)# permit ip  10.1.1.0 0.0.0.255 any
R2(config-ext-nacl)# exit

! Aplikace na sub-interfaces
R2(config)# interface GigabitEthernet0/0.100
R2(config-subif)# ip access-group VLAN100_OUT in
R2(config-subif)# exit
R2(config)# interface GigabitEthernet0/0.200
R2(config-subif)# ip access-group VLAN200_OUT in

! Ověření:
R2# show access-lists
// VYSVĚTLENÍ
  • deny je první → zákazová pravidla musí být před povoleními
  • permit ip ... any na konci zachovává ostatní komunikaci (inter-VLAN, DHCP, atd.)
  • Bez posledního permit by implicit deny zahodil i legitimní provoz
